Аналитический отчёт по проекту MP (AutoHub)
1. Критические уязвимости безопасности
1.1. Авторизация по коду (auth)
verify-code в production не проверяет код.
В auth.service.ts для номеров не из TEST_PHONES выполняется только поиск/создание пользователя по номеру телефона и выдача JWT. Поле code не используется.
Итог: зная номер жертвы, можно получить её JWT без SMS и без знания кода.

send-code ничего не отправляет.
Вызывается только нормализация и проверка длины номера. Нет интеграции с SMS-провайдером, нет хранения кода и лимитов.
Итог: «вход по коду» в production — фикция; по сути вход только по номеру телефона.

В dev в лог попадает информация о верификации:
console.log('[Auth] verifyCode', { phone: n, codeLength: code?.length }) — утечка метаданных авторизации.

1.2. IDOR и отсутствие проверки прав (Backend)
Организации (organizations.controller + organizations.service):

Нет проверки «пользователь принадлежит этой организации».
Любой авторизованный пользователь может:
GET /organizations/:id — читать любую организацию;
PATCH /organizations/:id — менять название, адрес, телефон, время и т.д.;
GET/POST /organizations/:id/staff, GET /organizations/:id/photos/:filename и т.д.
Итог: полный IDOR по организациям (чтение/изменение чужих СТО).
Заказы (orders.controller + orders.service):

Фото заказов:
GET/POST /orders/:orderId/photos, GET .../photos/:photoId/file не проверяют, что заказ принадлежит организации пользователя или клиенту по телефону. Достаточно знать orderId/photoId.
Итог: просмотр и загрузка фото чужих заказов.

Смена статуса и времени:
updateStatus(id, status) и updateTime(id, body) в сервисе не получают req.user и не проверяют принадлежность заказа.
Итог: любой авторизованный пользователь может менять статус и время любого заказа.

Назначение мастера:
assignMaster(id, masterId) тоже без проверки прав и принадлежности заказа к организации.
Итог: произвольное назначение мастера на любой заказ.

Позиции заказа:
PATCH /orders/:id/items передаёт в сервис только organizationId из req.user. У клиента organizationId === undefined, и в updateItems выполняется прямая перезапись позиций без проверки, что заказ принадлежит этому клиенту.
Итог: клиент может менять позиции любого заказа (не только своего).

Чаты (chats.controller + chats.service):

GET /chats/:id/messages — в контроллере не передаётся пользователь; сервис отдаёт все сообщения чата по chatId.
Итог: чтение переписки любого чата по известному ID.

POST /chats/:id/messages — в сервис передаётся только isFromClient (из того, есть ли у пользователя организация). Нет проверки, что чат принадлежит именно этой организации или этому клиенту (телефон).
Итог: можно писать в чужой чат и, в частности, отправлять запросы согласования от имени «СТО» в чат другой организации (создание заказов, смена статусов).

1.3. Path traversal (файлы)
organizations.service.ts — getPhotoPath(orgId, filename):
path.join(ORG_PHOTOS_DIR, orgId, filename) без проверки, что результирующий путь остаётся внутри ORG_PHOTOS_DIR/orgId.
Если в filename передать ../ или ..\, можно выйти за пределы директории организации и читать произвольные файлы с сервера.

Аналогично стоит проверить выдачу файлов фото заказов в orders.service (путь к файлу строится из сохранённого filePath и orderId); при любом использовании пользовательского filename без нормализации пути — тот же риск.

1.4. Загрузка файлов (Multer)
Нет ограничения размера и типа файла в FileInterceptor('file').
Риски: DoS большими файлами; загрузка исполняемых или опасных файлов под видом «фото».
1.5. CORS
app.enableCors() без параметров — разрешены любые origin.
В production лучше ограничить список разрешённых доменов.
1.6. Секреты и конфигурация
В .env.example: JWT_SECRET=your-super-secret-key-change-in-production — риск дефолтного секрета в production.
Нет rate limiting на /auth/send-code и /auth/verify-code — перебор номеров и брутфорс кодов (когда появятся) без ограничений.
2. Ошибки и согласованность API
2.1. throw new Error вместо HTTP-исключений
В нескольких местах используется throw new Error(...). Nest обрабатывает это как 500, а не 4xx:

chats.service.ts: throw new Error('client_phone required') — логичнее 400.
orders.controller.ts: throw new Error('Not found'), throw new Error('No file'), throw new Error('No organization') — лучше NotFoundException / BadRequestException.
organizations.controller.ts: throw new Error('Not found'), throw new Error('No file') — то же самое.
Итог: клиент получает 500 там, где ожидает 404/400; усложняется обработка ошибок на клиенте и различение сбоев сервера.

2.2. Нет единого формата ошибок
Нет глобального Exception Filter и единого формата тела ответа при ошибках (код, сообщение, requestId).
Клиентам сложнее единообразно обрабатывать ошибки.
2.3. Валидация входящих данных
Глобально включён ValidationPipe с whitelist: true, но DTO с class-validator по сути не используются; тела запросов типизированы как body: any или минимальные интерфейсы.
Нет явной валидации формата и диапазонов (телефон, даты, id, размер массивов и т.д.).
Итог: неверные или злонамеренные данные могут доходить до сервисов и БД; возможны массовое присвоение полей и неожиданное поведение.
3. Архитектура и консистентность
3.1. WebSocket
В backend нет реализации WebSocket (нет Gateway для /ws), при этом в Flutter в конфиге заданы wsPath = '/ws' и wsUrl.
Итог: при включённом enableWs клиенты будут пытаться подключиться к несуществующему эндпоинту; реальные обновления только через REST.
3.2. Refresh token (client2)
В autohub_client2 есть логика обновления сессии при 401 (refreshSession, _Refresh401Interceptor), ожидается refresh_token и эндпоинт обновления.
Backend отдаёт только access_token, отдельного refresh-эндпоинта нет.
Итог: после истечения JWT (например, 7d) при 401 refresh не сработает, пользователь окажется разлогинен без явного «сессия истекла».
3.3. Хранение токена (Flutter)
Токен хранится в SharedPreferences (и в business, и в client2). На Android это не шифрованное хранилище; при root/бэкапах токен может быть извлечён.
Для чувствительных сценариев предпочтительнее flutter_secure_storage (или аналог).
3.4. Каталог (catalog)
Эндпоинты каталога защищены JWT (@UseGuards(AuthGuard('jwt'))).
Если по задумке поиск СТО должен быть доступен до входа — нужен публичный каталог; если только для авторизованных — текущее поведение согласовано, но это нужно явно зафиксировать.
4. База данных и окружение
4.1. TypeORM
synchronize: process.env.NODE_ENV !== 'production' — в dev схема подстраивается под сущности.
В production предполагаются только миграции. Риск: если когда-либо запустят без NODE_ENV=production или с ошибкой в env, возможна автоматическая перезапись схемы в проде.
4.2. Миграции
Есть папка миграций и скрипты migration:generate / migration:run.
Нет явного описания в README/документации, что в production запрещён synchronize и обязательны миграции — риск расхождения схемы и потери данных при ручных действиях.
4.3. Сид
Seed выполняется при старте приложения (OnApplicationBootstrap) в non-production или при SEED_DEV.
Нужно гарантировать, что в production SEED_DEV не установлен и seed не выполняется, иначе возможна перезапись/дозапись тестовых данных.
5. Надёжность и наблюдаемость
5.1. Логирование
Только console.log / console.error в разных модулях (auth, orders, chats, booking, seed и т.д.).
Нет структурированного логгера (уровни, JSON, контекст запроса), нет ротации и централизованного сбора логов.
5.2. Мониторинг
Нет health-check эндпоинта (/health, /ready), метрик и трейсинга.
В production сложно автоматически проверять готовность сервиса и диагностировать сбои.
5.3. Обработка ошибок в чатах
В chats.controller.ts при ошибке getChatsForClient ловится исключение, в ответ возвращается { items: [] }.
Ошибка только пишется в консоль; клиент не отличает «чатов нет» от «произошла ошибка» — можно скрыть сбой.
6. Тесты и деплой
6.1. Backend
В backend нет тестов (нет *.spec.ts/*.test.ts, в package.json нет скриптов test/jest/vitest).
Регрессии при рефакторинге и при исправлении уязвимостей не отлавливаются автоматически.
6.2. Деплой
Нет Dockerfile / docker-compose в репозитории, нет описанных CI/CD (например, GitHub Actions).
Деплой и окружения зависят от ручных действий и локальной конфигурации; возможны расхождения между средами.
