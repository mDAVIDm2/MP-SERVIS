# Доступность слотов и когда подключать БД

## Логика слотов (текущая)

- Слоты СТО — ячейки по **30 минут** (09:00, 09:30, …, 17:30).
- Суммарное время услуг (например 3 ч) должно помещаться **одним непрерывным блоком**.
- Если в этом блоке хотя бы один слот занят — время начала недоступно.
  - Пример: 11:30 занято, нужно 3 ч → 11:00 выбрать нельзя (блок 11:00–14:00 захватил бы 11:30).
- Занятые интервалы задаются в моках (`MockData.getBusyRangesForStoOnDate`) и при переходе на бэкенд придут из API `catalog/organizations/:id/availability`.

## Разбить на несколько визитов

Задел: пользователь сможет разбить время на весь день (несколько визитов).
Сейчас в UI есть подсказка; логику нескольких непересекающихся блоков с той же суммой длительностей можно добавить поверх текущего `availability_helper` (например, подбор комбинаций свободных окон).

## Когда вводить БД

**Сейчас** достаточно моков: сценарии записи и отображения слотов уже проверяются.

Имеет смысл вводить БД (и бэкенд), когда:

- появятся реальные пользователи и нужно хранить записи, заказы, чаты;
- нужно единое расписание по всем СТО и датам;
- понадобится синхронизация между устройствами и офлайн.

Рекомендация: **сначала зафиксировать API** (эндпоинты в `api_endpoints.dart`, форматы ответов), затем реализовать бэкенд и заменить моки вызовами API. Структуры (слоты, занятость, длительности) уже заложены в `availability_helper` и репозиториях.
