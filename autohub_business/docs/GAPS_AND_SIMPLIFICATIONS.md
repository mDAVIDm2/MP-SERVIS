# Пробелы и упрощения AutoHub Business

Сверка с промптом и текущим кодом. Что не реализовано, что упрощено, что нужно доработать.

---

## 1. Не реализовано

### 1.1 Бэкенд и данные
- **Бэкенд** — по плану (NestJS, PostgreSQL, Redis, S3, WebSocket) не создан; приложение работает с локальными репозиториями и при недоступности API подставляет демо-данные.
- **Оплата** — в промпте упомянута «оплата» в карточке заказа; в приложении нет экранов/полей оплаты и привязки к заказу.

### 1.2 Чаты — реализовано с API
- **ChatRepository** использует ChatApiService и WebSocket: загрузка чатов/сообщений с API, отправка сообщений и запросов согласования; оптимистичное обновление; при ошибке — SnackBar. Демо-данные только при ошибке API и пустом списке.

### 1.3 Организация, персонал, настройки — реализовано с API
- **Организация** — загрузка/сохранение через `GET/PATCH /organizations/:id`; кэш в SharedPreferences; pull-to-refresh.
- **Персонал** — загрузка/приглашение/обновление/деактивация через `GET/POST/PATCH /organizations/:id/staff`; кэш в prefs; pull-to-refresh.
- **Настройки** — загрузка/сохранение через `GET/PATCH /organizations/:id/settings`; кэш в prefs; pull-to-refresh на экране.

### 1.4 Заказы — интеграция с API
- **completeOrderItem** — реализован через `PATCH /orders/:id/items` в OrderApiService; вызов из OrderRepository; тап по позиции в детали заказа и в «Мои задачи»; при ошибке — SnackBar.
- **Загрузка одного заказа по id** — при открытии детали используется кэш из репозитория; отдельного `GET /orders/:id` при открытии экрана нет (данные есть после loadFromApi или прихода по WS).

### 1.5 Мастер — реализовано
- **Фото по заказу** — экран с загрузкой с API, добавление фото (image_picker + POST), отображение через авторизованный запрос.
- **Доп. работы** — экран с добавлением позиции через `addExtraWorkItem` (PATCH items); SnackBar при успехе/ошибке.

### 1.6 Уведомления и push
- Нет экранов/логики для **push-уведомлений** (регистрация токена, список уведомлений).
- ApiEndpoints есть `notifications`, `registerPushToken` — в приложении не используются.

### 1.7 Аналитика и финансы (по промпту)
- В промпте для Owner: «Аналитика и финансы», «выручка за период, средний чек». В приложении: только «заказов сегодня», «выручка сегодня», «по сотрудникам (сегодня)» — без выбора периода и без среднего чека.

## 2. Упрощённый функционал

### 2.1 Авторизация
- **Реальный API** вызывается (sendSmsCode, verifyCode); при ошибке сети или отсутствии бэкенда остаётся **демо-вход** по кодам 1111/2222/3333 и любому 4-значному как Admin.
- **Refresh token** — не реализован (есть только в ApiEndpoints); при истечении access token логика обновления не делается.
- **Профиль с бэкенда** — после входа данные берутся из ответа verify или из prefs; отдельный запрос GET /profile при старте приложения не выполняется (можно добавить для актуализации роли/организации).

### 2.2 Заказы
- **Мутации** (setOrderStatus, assignMaster, cancelOrder) — возвращают `Result`; при ошибке API выполняется откат state и в UI показывается SnackBar.
- **Создание заказа** — при ошибке API заказ добавляется локально (без id с сервера); дубликаты при повторной отправке возможны.

### 2.3 Согласование доп. работ в чате
- Карточка запроса согласования и кнопки «Согласовать» / «Отклонить» в UI есть. Ответ клиента **уходит на бэкенд** (POST `/orders/:id/approval`, POST `/orders/:id/confirm`); бэкенд фильтрует добавляемые позиции по `approvedItemIds`.

### 2.4 Клиенты
- Список клиентов строится **только из заказов** (агрегация по имени+телефону); отдельного API/сущности «клиенты СТО» нет (в плане бэкенда — возможно через organizationClients).

### 2.5 Календарь
- Вид «Неделя» — полоска дней и счётчик заказов; **слоты по времени** и «привязка заказов к мастерам и времени» в календаре упрощены (нет сетки слотов и перетаскивания).

### 2.6 Роутинг
- **go_router** настроен с redirect по auth и роли; вложенных маршрутов для экранов деталей (например `/app/orders/:id`) нет — переход на OrderDetailScreen и др. по-прежнему через `Navigator.push`, а не через `context.push('/app/orders/$id')`. Deep link и восстановление стека при перезапуске не прорабатывались.

## 3. Что доработать в первую очередь

**Выполнено (шаги 1–7):**
- Чаты — ChatRepository на API + WebSocket; оптимистичные обновления; SnackBar при ошибке.
- completeOrderItem — PATCH orderItems в API; тап по позиции в детали и в «Мои задачи».
- Мастер: Фото и Доп. работы — экраны и API (orderPhotos, addExtraWorkItem).
- Обработка ошибок API для заказов — мутации с Result, откат state, SnackBar.
- Организация и персонал с API — загрузка/сохранение через API; pull-to-refresh.
- Настройки с API — загрузка/сохранение через API; pull-to-refresh.
- IMPLEMENTATION_STATUS.md и GAPS_AND_SIMPLIFICATIONS.md обновлены.

**Дальнейшие улучшения:**
- Оплата, push-уведомления, аналитика по периоду, refresh token, GET /profile при старте.
- Детальные маршруты (go_router для `/app/orders/:id` и т.п.).
