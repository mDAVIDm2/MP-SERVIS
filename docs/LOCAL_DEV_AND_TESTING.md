# Запуск и тестирование экосистемы MP-Servis

Один бэкенд (NestJS + PostgreSQL), два приложения: **MP-Servis Business** (СТО/мастера) и клиент **MP-Servis**. Оба подключаются к одному API и одной БД.

---

## 1. Требования

- **Node.js** 18+
- **PostgreSQL** 14+ (локально или Docker)
- **Flutter** 3.10+
- **Dart** 3.10+

---

## 2. База данных

### Вариант A: Локальный PostgreSQL

1. Установите PostgreSQL и создайте БД:
   ```bash
   createdb mp_servis
   ```
2. Либо через psql:
   ```sql
   CREATE DATABASE mp_servis;
   ```

### Вариант B: Docker

```bash
docker run -d --name mp-servis-pg -e POSTGRES_PASSWORD=postgres -e POSTGRES_DB=mp_servis -p 5432:5432 postgres:16-alpine
```

Подключение: `postgresql://postgres:postgres@localhost:5432/mp_servis`

---

## 3. Бэкенд

```bash
cd backend
cp .env.example .env
# При необходимости отредактируйте .env (DATABASE_URL, PORT, JWT_SECRET)

npm install
npm run start:dev
```

- API: **http://localhost:3000/api/v1**
- При первом запуске TypeORM создаст таблицы (synchronize в dev).
- В логах должно быть: `MP-Servis API: http://localhost:3000/api/v1`

---

## 4. Приложения Flutter

Оба приложения по умолчанию обращаются к `http://localhost:3000/api/v1` (и `ws://localhost:3000/ws` для Business). Для эмулятора/телефона замените `localhost` на IP вашего компьютера в:
- **Business:** `autohub_business/lib/core/api/api_endpoints.dart`
- **Client:** `autohub_client2/lib/core/api/api_endpoints.dart`

### MP-Servis Business (СТО)

```bash
cd autohub_business
flutter pub get
flutter run
```

- Вход: реальный API (send-code → verify-code). При ошибке сети — демо-коды: `1111` (мастер), `2222` (владелец), `3333` (самозанятый), любой другой 4-значный — администратор.
- У владельца/админа: заказы, чаты, календарь, организация, персонал, настройки — с API.

### MP-Servis (клиент)

```bash
cd autohub_client2
flutter pub get
flutter run
```

- Вход: реальный API. Номер телефона — любой (например `+7 999 123-45-67`), код — любой 4-значный (бэкенд в dev не проверяет SMS).
- После входа клиент видит **свои заказы** (где в заказе указан его телефон) и **чаты** по этим заказам.

## 5. Сценарий тестирования связки

1. **Запустите** PostgreSQL, бэкенд, затем Business и Client (в двух окнах/устройствах или переключая приложения).

2. **Business (владелец/админ):**
   - Войдите (например демо-код `2222` или реальный API).
   - Создайте заказ из календаря (FAB → форма): укажите **телефон клиента** в формате, который потом будете вводить в Client (например `79991234567` или `+7 999 123-45-67`), имя клиента, авто, услуги, дату.
   - Сохраните. Появится заказ и чат по заказу (чат создаётся автоматически).

3. **Client:**
   - Войдите с **тем же номером телефона** (любой код, например `1234`). Бэкенд найдёт или создаст пользователя по телефону.
   - В разделе заказов (или «Мои заказы») должен отобразиться созданный заказ.
   - В разделе чатов — чат по этому заказу. Отправьте сообщение — оно сохранится в БД.

4. **Business:**
   - Обновите список чатов/сообщений — сообщение от клиента должно быть видно (при необходимости pull-to-refresh).
   - Подтвердите заказ, назначьте мастера — статус изменится.

5. **Client:**
   - Обновите заказ/чат — должны отображаться обновлённый статус и ответы от СТО.

## 6. Полезные моменты

- **Нормализация телефона:** бэкенд сравнивает телефоны по цифрам (все нецифровые символы отбрасываются). Клиент видит заказы, где `client_phone` в БД совпадает с его номером (по цифрам).
- **Роли:** пользователь без `organization_id` в БД считается клиентом (GET /orders и GET /chats возвращают данные по его телефону). Пользователь с `organization_id` — сотрудник СТО (видит заказы/чаты своей организации).
- **Чаты:** создаются автоматически при создании заказа. Сообщения из Client помечаются `is_from_client: true`.
- Если что-то не отображается — проверьте, что бэкенд запущен и в приложениях указан правильный baseUrl (для устройства не localhost).

## 7. Структура репозитория

```
MP/
├── backend/          # NestJS API, TypeORM, PostgreSQL
├── autohub_business/ # Flutter — СТО и мастера
├── autohub_client2/  # Flutter — клиенты
└── docs/             # Промпты и инструкции
```

Оба приложения используют один и тот же набор endpoints (auth, orders, chats и т.д.); различие только в том, что возвращает API в зависимости от роли пользователя (организация vs клиент по телефону).
