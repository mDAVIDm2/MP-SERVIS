# MASTER PROMPT: AUTOHUB BUSINESS — ПРИЛОЖЕНИЕ ДЛЯ СТО И ЧАСТНЫХ МАСТЕРОВ

Единый документ для разработки приложения **AutoHub Business**: мобильное приложение (Android), веб-кабинет и десктоп. Всё — одна экосистема с уже готовым **AutoHub Client** (клиентское приложение).

---

## ЧТО ДЕЛАТЬ ДАЛЬШЕ (ПРОСТЫМИ СЛОВАМИ)

1. **Создать рядом с клиентским приложением новый проект** — `autohub_business` (или отдельная папка в том же репозитории). Не трогать `autohub_client`.
2. **Сначала сделать мобильное приложение под Android** — полный функционал для СТО и мастеров.
3. **Потом из этого же проекта Flutter** собрать веб-сайт (Flutter Web) и приложение для Windows/macOS (Flutter Desktop). Один код — три платформы.
4. **Одна база данных и один бэкенд** — и клиентское приложение, и Business ходят в одни и те же API. Общие сущности (Order, Car, User, статусы заказов) должны называться одинаково в обоих приложениях — лучше вынести в общий пакет `autohub_shared` позже.
5. **Интерфейс зависит от роли** — кто вошёл (директор, администратор, мастер или самозанятый), такие вкладки и экраны он и видит. Лишнего не показывать.

---

## СТРАТЕГИЯ ПЛАТФОРМ

| Этап | Платформа | Как |
|------|-----------|-----|
| 1 | **Android** (основной сейчас) | Flutter-приложение, мобильный UI |
| 2 | **Сайт (Web)** | Тот же Flutter-проект → `flutter build web` |
| 3 | **Десктоп (Windows/macOS)** | Тот же Flutter-проект → `flutter build windows` / `macos` |

Один репозиторий `autohub_business`, одна кодовая база. Адаптивная вёрстка: на телефоне — нижняя панель и списки, на сайте и десктопе — боковое меню, таблицы, широкий календарь.

## ЭКОСИСТЕМА: КАК ВСЁ СВЯЗАНО

```
[ AutoHub Client ]     [ AutoHub Business ]
   (готово)                 (создать)
       │                          │
       └──────────┬───────────────┘
                  ▼
         [ Единый бэкенд ]
         API: /api/v1/*
         WebSocket, одна БД
```

- **Клиент** записывается в приложении Client → заказ создаётся в БД → **Business** видит новую заявку.
- **Админ в Business** подтверждает заказ, назначает мастера → **Клиент** видит обновление статуса.
- **Мастер** отмечает выполнение → статус меняется → **Клиент** видит прогресс.
- **Чаты и согласование доп. работ** — общие: и в Client, и в Business один и тот же чат по заказу.

Общие правила:
- Один бэкенд, одни и те же поля в API (orderNumber, carId, status и т.д.).
- Статусы заказа (pendingConfirmation, confirmed, inProgress, done, cancelled) — одинаковые названия в обоих приложениях.
- По возможности общие модели в пакете `autohub_shared`, чтобы при изменении поля не ломать второе приложение.

## РОЛИ И ДИНАМИЧЕСКИЙ ИНТЕРФЕЙС (ROLE-BASED UI)

Интерфейс **строго зависит от роли**. После входа по токену с бэкенда приходит роль — показываем только разрешённые разделы.

---

### 1. ДИРЕКТОР / ВЛАДЕЛЕЦ (OWNER)

**Задача:** Контроль бизнеса, цифры, аналитика, финансы.

- **Главный экран:** Дашборд с аналитикой (выручка за период, средний чек, количество заказов), **загрузка по сотрудникам** (кто сколько заказов выполнил / принёс выручки), KPI сервиса.
- **Навигация (нижняя панель или боковое меню):**  
  Дашборд | Аналитика и финансы | Заказы | Персонал | Настройки
- **Видит:** Всё по своей организации (или по всем своим организациям, если их несколько). Управление услугами, ценами, персоналом, удаление организации (только владелец).

---

### 2. АДМИНИСТРАТОР (ADMIN)

**Задача:** Ежедневная работа с клиентами: заказы, диалоги, распределение задач по мастерам.

- **Главный экран:** Заказы на сегодня или канбан по статусам; блок «Требуют внимания» (новые заявки, ожидают согласования).
- **Навигация:**  
  Заказы | **Диалоги (Чаты)** | Календарь | Клиенты | Профиль
- **Ключевое:** Раздел **Диалоги** на первом плане — общение с клиентами. В Заказах/Календаре — **закрепление заказов за мастерами** (назначение мастера на заказ). Видит цены, телефоны, чаты. Не может удалять организацию и менять владельца.

---

### 3. МАСТЕР В СТО (MASTER)

**Задача:** Видеть свои задачи на день и отчитываться о выполнении.

- **Главный экран:** **Календарь / расписание на день** или список «Мои заказы» на сегодня.
- **Навигация (только 2 пункта):**  
  Календарь (или Мои заказы) | Профиль
- **Ограничения (обязательно):**
  - **Не видит цены** (API отдаёт null для полей с ценой).
  - **Не видит телефоны клиентов** (только марка/модель авто, гос. номер, комментарий).
  - **Нет вкладки Диалоги** — с клиентами общается только админ.
  - Видит только заказы, **назначенные на него** (master_id = текущий пользователь).
- В карточке задачи: чек-лист работ, кнопки «Добавить фото», «Запросить доп. работы» (запрос уходит админу, цену ставит админ).

---

### 4. ИНДИВИДУАЛЬНЫЙ СПЕЦИАЛИСТ / САМОЗАНЯТЫЙ (SOLO)

**Задача:** Один человек и владелец, и исполнитель. Без раздела «Персонал» и без назначения задач другим.

- **Интерфейс:** Объединённый режим «как админ + как мастер», но **без меню «Персонал»** и без выбора «назначить мастера» (всегда сам).
- **Навигация:**  
  Календарь | Заказы | Диалоги | Настройки (услуги, цены, профиль)
- Сам ведёт чаты, сам выставляет цены, сам отмечает выполнение.

---

## ОБЩАЯ НАВИГАЦИЯ ПО РОЛЯМ (СВОДКА)

| Раздел        | Owner | Admin | Master | Solo |
|---------------|-------|-------|--------|------|
| Дашборд/аналитика | ✓     | —     | —      | —    |
| Заказы        | ✓     | ✓     | ✓ (только свои) | ✓    |
| Календарь     | ✓     | ✓     | ✓ (свои) | ✓    |
| Диалоги (чаты)| ✓     | ✓     | ✗      | ✓    |
| Клиенты      | ✓     | ✓     | ✗      | ✓    |
| Персонал     | ✓     | ✓     | ✗      | ✗    |
| Финансы/аналитика | ✓ | ✓   | ✗      | ✓ (своя) |
| Настройки (услуги, цены) | ✓ | ✓ | ограниченно | ✓ |

## КЛЮЧЕВЫЕ ЭКРАНЫ (КРАТКО)

- **Авторизация** — вход по телефону + SMS (или логин/пароль, как решит бэкенд); после входа по роли показываем нужный главный экран и набор вкладок.
- **Дашборд (Owner/Admin)** — KPI, загрузка по сотрудникам, «Требуют внимания», заказы на сегодня.
- **Заказы** — список: активные / история; фильтры; карточка заказа (номер, клиент, авто, услуги, статус, сумма); детали заказа + смена статуса, назначение мастера, оплата.
- **Календарь** — день/неделя; слоты по времени; закрепление заказов за мастерами; создание заказа; для мастера — только свои слоты.
- **Диалоги (Чаты)** — список чатов с клиентами; экран чата; запрос согласования доп. работ (карточка с ценами уходит клиенту).
- **Клиенты** — поиск, список клиентов СТО, карточка (контакты, авто, история заказов **только в этой СТО**), внутренние заметки.
- **Персонал** — список, приглашение (ссылка/код), роли, деактивация (только Owner/Admin).
- **Мои задачи (Master)** — текущий заказ (без цен и телефона), чек-лист работ, «Добавить фото», «Запросить доп. работы», «Завершить»; список следующих задач.
- **Настройки** — профиль организации (адрес, график, логотип), услуги и цены, специализация по маркам, уведомления, шаблоны сообщений.

## STATE MACHINE ЗАКАЗА (ЕДИНАЯ ЛОГИКА С КЛИЕНТОМ)

Статусы и переходы те же, что в клиентском приложении (бэкенд один):

1. **PENDING_CONFIRMATION** — новая заявка от клиента → Админ подтверждает или отклоняет.
2. **CONFIRMED** — Админ назначает мастера; можно перевести «В работу».
3. **IN_PROGRESS** — мастер выполняет; при необходимости доп. работ Админ отправляет согласование → **PENDING_APPROVAL**.
4. **PENDING_APPROVAL** — ожидание ответа клиента в приложении Client; после ответа — снова **IN_PROGRESS** (или учёт отклонённого).
5. **COMPLETED** — работы завершены, ожидание выдачи/оплаты.
6. **DONE** — авто выдано, оплата зафиксирована.
7. **CANCELLED** — отменён.

Цвета статусов — как в клиенте (жёлтый, синий, зелёный, красный), чтобы везде было одинаково.

## ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ

- **Язык:** Flutter (Dart). Один проект для Android, Web и Desktop.
- **State management:** Riverpod (как в Client).
- **Роутинг:** go_router (или аналог) с проверкой роли: при заходе на запрещённый раздел — редирект на разрешённый главный экран.
- **API:** Dio, те же базовые URL и `/api/v1/*`. В заголовках — JWT (и при необходимости organization_id для Owner).
- **Real-time:** WebSocket для новых заявок, смены статусов, сообщений в чате.
- **Тема:** Тёмная. Фон #0D0D0D, #1A1A1A; акцент **#FF6B00** (оранжевый). Тексты #FFFFFF, #999999, #666666.
- **Структура проекта:** По аналогии с `autohub_client`: `lib/core/`, `lib/features/`, `lib/shared/`. В `core` — api, auth, theme, router; в `features` — по фичам (dashboard, orders, calendar, chats, profile, master_tasks, settings); в `shared` — общие виджеты, модели.

## ИНСТРУКЦИЯ ДЛЯ ПРОДОЛЖЕНИЯ РАЗРАБОТКИ

Используй это как пошаговую инструкцию для себя или для нейросети.

### Шаг 1. Создать проект autohub_business

- В **той же папке**, где лежит `autohub_client` (например `c:\dev\`), создать новый Flutter-проект:  
  `flutter create autohub_business`
- Или внутри одного репозитория завести папку `autohub_business` и там инициализировать Flutter (как удобнее для монорепо).

### Шаг 2. Скопировать подход из клиента

- Посмотреть структуру `autohub_client/lib`: `core/`, `features/`, `shared/`, `main.dart`.
- В `autohub_business` сделать такую же структуру: `core/` (api, auth, theme, router, constants), `features/` (auth, dashboard, orders, calendar, chats, clients, staff, settings, master_tasks), `shared/` (widgets, models).
- Подключить Riverpod, go_router, Dio по аналогии с клиентом.

### Шаг 3. Роутинг и роли

- После авторизации сохранять в состоянии не только токен, но и **роль** (owner / admin / master / solo).
- Главный экран после входа выбирать по роли:
  - Owner → Дашборд (аналитика).
  - Admin → Заказы или Диалоги (на выбор как стартовый).
  - Master → Календарь или Мои заказы.
  - Solo → Календарь или Заказы.
- Нижнюю панель (или боковое меню на Web/Desktop) строить динамически: только те пункты, которые разрешены для текущей роли.

### Шаг 4. Экраны по приоритету (MVP)

1. Авторизация (экран входа, сохранение токена и роли).
2. Оболочка с навигацией по ролям (разные наборы вкладок).
3. Заказы: список (активные/история), детали заказа, смена статуса, назначение мастера (для Admin/Owner).
4. Для Master: экран «Мои задачи» (текущий заказ, чек-лист, фото, запрос доп. работ).
5. Диалоги (Чаты) для Admin/Owner/Solo: список чатов, экран чата, отправка сообщений, запрос согласования.
6. Дашборд для Owner: KPI, загрузка по сотрудникам.
7. Календарь: вид на день/неделю, привязка заказов к мастерам и времени.
8. Настройки: профиль организации, услуги и цены (базово).

### Шаг 5. Сборка Web и Desktop

- После стабилизации Android:  
  `flutter build web`  
  `flutter build windows` / `flutter build macos`
- Проверить адаптивность: на узком экране — мобильный вид (нижняя панель), на широком — боковое меню, таблицы, широкий календарь.

### Шаг 6. Общая база и контракт (когда будет бэкенд)

- Подключить оба приложения к одному API.
- Имена полей и статусов заказа делать **идентичными** в Client и Business (лучше общий пакет `autohub_shared` с моделями Order, OrderStatus, Car, User и т.д.).
