# ПРОМТ: AUTOHUB STO — МОБИЛЬНОЕ ПРИЛОЖЕНИЕ И САЙТ ДЛЯ АВТОСЕРВИСОВ

> **Актуальный единый промт:** см. [AUTOHUB_BUSINESS_MASTER_PROMPT.md](./AUTOHUB_BUSINESS_MASTER_PROMPT.md) — там объединены роли (Owner/Admin/Master/Solo), стратегия Android → Web → Desktop и пошаговая инструкция для создания `autohub_business`.

## Цель документа
Спецификация для разработки **мобильного приложения для СТО** (Android, опционально iOS) и **веб-сайта/кабинета СТО**, работающих с **общим бэкендом** экосистемы AutoHub. Клиентское приложение (AutoHub Client) уже реализовано; данное приложение — для владельцев и сотрудников автосервисов.

---

## Связь с экосистемой

- **Один бэкенд**: NestJS + PostgreSQL + Redis + S3 + WebSocket (см. Master Prompt, Часть 4).
- **Общие модели данных**: User, Organization, Order, OrderItem, Car, Chat, Review и т.д. — те же сущности и статусы (State Machine заказа без изменений).
- **Изоляция данных**: СТО видит только свои заказы, своих клиентов, своих сотрудников. Клиент видит только свои данные. Перекрёстный доступ между СТО запрещён.
- **Совместная работа**: клиент записывается в приложении Client → заказ появляется в приложении STO; смена статуса в STO → клиент видит обновление; чат и согласование доп. работ — общие.

## Роли (как в Master Prompt)

- **OWNER** — владелец; может несколько организаций; полный доступ.
- **ADMIN** — администратор одной организации; заказы, календарь, чаты, клиенты, персонал, финансы, настройки (без удаления организации).
- **MASTER** — мастер; только свои задачи; **без цен**, **без телефонов клиентов**, без чатов с клиентами; может отмечать выполнение, добавлять фото, запрашивать доп. работы.

## Язык и дизайн

- **Язык интерфейса**: русский (все тексты, кнопки, ошибки, push).
- **Тема**: тёмная, профессиональная. Цвета из Business Prompt:
  - Фон: #0D0D0D, #1A1A1A, #252525.
  - Акцент: #FF6B00 (оранжевый).
  - Текст: #FFFFFF, #999999, #666666.
  - Статусы заказов: как в Client (PENDING_CONFIRMATION — жёлтый, CONFIRMED/IN_PROGRESS — синий, COMPLETED/DONE — зелёный, CANCELLED — красный).

## Навигация (мобильное приложение СТО)

Нижняя панель (аналог клиентского приложения, но пункты другие):

1. **Главная** — дашборд: заказы сегодня, требуют внимания, KPI.
2. **Календарь** — день/неделя: записи по времени (упрощённый ресурсный вид или список по дням).
3. **Заказы** — список заказов (активные / история), фильтры, поиск.
4. **Чаты** — чаты с клиентами (только Owner/Admin; у Master этого пункта нет).
5. **Профиль** — организация, сотрудники, настройки, выход.

Для роли **Master** внизу только: **Мои задачи** и **Профиль**.

---

## Ключевые экраны (кратко)

- **Дашборд**: KPI (заказов сегодня, выручка, среднее время, рейтинг); блок «Требуют внимания» (новые заявки, согласование, SLA); заказы на сегодня списком.
- **Календарь**: вид «День» или «Неделя», список заказов по времени; тап — детали заказа; создание заказа; смена статуса, назначение мастера.
- **Заказы**: вкладки Активные / История; фильтры (статус, даты, мастер); карточки заказов (номер, клиент, авто, услуги, статус, сумма); детальный экран заказа с действиями по статусу.
- **Чаты**: список чатов с клиентами; экран чата (сообщения, шаблоны, запрос согласования); без доступа для Master.
- **Клиенты** (в профиле или отдельная вкладка): поиск, список клиентов СТО; карточка клиента — контакты, авто, история заказов **только в этой СТО**, внутренние заметки.
- **Персонал** (Owner/Admin): список сотрудников, приглашение (телефон/email, роль), редактирование, деактивация.
- **Настройки**: профиль организации (название, адрес, телефон, часы работы, логотип); услуги и цены (категории, позиции с ценой и длительностью); специализация по маркам; слоты и таймаут подтверждения; уведомления; шаблоны сообщений.
- **Мои задачи** (только Master): текущая задача (авто, работы без цен, чеклист выполнения, комментарий клиента); кнопки: добавить фото, запрос доп. работ, завершить; список следующих задач.

## API и данные

- Использовать **те же endpoints**, что и в Master Prompt: авторизация, организации, заказы, каталог, чаты, уведомления и т.д.
- Все запросы **scoped по organization_id** (из JWT или выбора организации).
- Для роли Master: API не возвращает цены и телефоны клиентов; список заказов — только с `master_id = current_user`.

## Общая структура БД и совместимость с клиентом

Чтобы в будущем нейросеть или разработчик мог уверенно связывать оба приложения с одной БД:

1. **Не дублировать определения сущностей**  
   Иметь один источник правды: либо бэкенд (модели + API), либо общий пакет (например `autohub_shared` с DTO и enum). И клиент, и СТО используют одни и те же имена полей и статусов.

2. **Имена полей и enum — идентичные**  
   Пример: `OrderStatus.done`, `OrderStatus.pendingConfirmation`, поля `orderNumber`, `carId`, `stoId`, `odometerAtCompletion` и т.д. В клиенте и в СТО — одни и те же ключи в JSON и в коде.

3. **Версионирование API**  
   Один базовый путь `/api/v1/`. Новые поля добавлять как опциональные, чтобы старые версии приложений не падали.

4. **Рекомендуемая структура репозитория (когда будет два приложения)**  
   - `autohub_backend` — API и БД.  
   - `autohub_shared` (пакет или подпапка) — общие модели, enum, константы (в т.ч. статусы заказа, форматы дат/денег).  
   - `autohub_client` — текущее клиентское приложение.  
   - `autohub_sto` — мобильное приложение для СТО (и при необходимости веб-кабинет).

Тогда «связать два приложения с одной БД» = оба подключаются к одному бэкенду и используют общий контракт (и по возможности общий пакет моделей).

## Что взять из текущего клиентского проекта

- **Структура `lib/`**: `core/` (api, auth, theme, constants), `features/` по фичам, `shared/` (виджеты, модели).
- **State management**: Riverpod (как в клиенте).
- **Навигация**: тот же подход (нижняя панель + стеки по вкладкам).
- **Тема и цвета**: отдельный `AppTheme` для СТО с оранжевым акцентом (#FF6B00), но те же принципы (тёмный фон, карточки, статусы).
- **Форматирование**: те же правила для денег (копейки → рубли), дат, телефонов (общие утилиты или общий пакет).

## Минимальный MVP для СТО (мобильное приложение)

- Авторизация (телефон + SMS или логин/пароль по решению бэкенда).
- Дашборд: заказы на сегодня, счётчики.
- Список заказов: активные/история, детали заказа, смена статуса (подтвердить, в работу, завершить, отмена).
- Деталь заказа: клиент, авто, услуги, сумма, комментарий; кнопки по статусу (подтвердить, начать, завершить, зарегистрировать оплату).
- Чаты с клиентами (Owner/Admin): список чатов, экран чата, отправка сообщения, запрос согласования доп. работ.
- Для Master: экран «Мои задачи» (текущий заказ, чеклист работ без цен, фото, запрос доп. работ, завершить).
- Настройки: профиль организации, услуги и цены, уведомления.
- Push-уведомления: новая заявка, сообщение в чате, ответ по согласованию.

## Как передать контекст ассистенту для реализации

1. Открыть в Cursor **корень проекта** (где лежит клиентское приложение или монорепо).
2. Положить этот промпт в репозиторий (например `docs/AUTOHUB_STO_PROMPT.md`).
3. Написать в чате:  
   «Нужно сделать приложение для СТО по спецификации из `docs/AUTOHUB_STO_PROMPT.md`. По структуре и стилю ориентироваться на текущий клиентский проект (те же core/features/shared, Riverpod, те же принципы API). Создай проект `autohub_sto` (или папку в монорепо) и реализуй экраны: авторизация, дашборд, список заказов, детали заказа, чаты, мои задачи для мастера, настройки.»

При необходимости можно добавить в этот же файл детали по экранам (макеты, поля форм) по мере уточнения требований.
